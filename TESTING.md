# 语音双向通话测试指南

## 快速测试

### 1. 启动服务器
```bash
npm start
```

服务器将在 http://localhost:4001 启动

### 2. 打开两个浏览器窗口

**窗口 A（发送者）：**
1. 访问 http://localhost:4001
2. 输入房间 ID（例如：`test-room`）
3. 输入用户名（例如：`Alice`）
4. 点击"加入房间"
5. 允许麦克风权限
6. 点击"开始说话"按钮
7. 对着麦克风说话

**窗口 B（接收者）：**
1. 访问 http://localhost:4001
2. 输入**相同的房间 ID**（`test-room`）
3. 输入不同的用户名（例如：`Bob`）
4. 点击"加入房间"
5. 允许麦克风权限
6. 应该能听到窗口 A 的声音

### 3. 双向测试

- 在窗口 B 也点击"开始说话"
- 两个窗口应该能互相听到对方的声音
- 这就是**双向通话**！

## 多人测试

可以打开 3+ 个窗口，所有人加入同一个房间：
- 每个人都能听到其他所有人的声音
- 这是一个**多人语音聊天室**

## 功能验证清单

### ✅ 基础功能
- [ ] 麦克风权限请求成功
- [ ] 加入房间成功（显示"已加入房间"）
- [ ] 看到其他用户加入提示
- [ ] 点击"开始说话"后按钮变为"停止说话"

### ✅ 音频传输
- [ ] 说话时能看到音频波形动画
- [ ] 对方能听到自己的声音
- [ ] 声音清晰，无明显失真
- [ ] 延迟低于 200ms（几乎实时）

### ✅ 静音检测
- [ ] 不说话时，网络流量很小（DTX 生效）
- [ ] 说话时，网络流量增加
- [ ] 静音时对方听不到噪音

### ✅ 网络健壮性
- [ ] 短暂断网后自动重连
- [ ] 丢包时声音仍然连续（FEC 生效）
- [ ] 多人同时说话时不卡顿

### ✅ 浏览器兼容性
- [ ] Chrome/Edge（推荐）
- [ ] Firefox
- [ ] Safari（需要 HTTPS 或 localhost）

## 常见问题

### 1. 听不到声音
- 检查浏览器是否允许麦克风权限
- 检查系统音量和麦克风音量
- 确保两个窗口在**同一个房间**
- 打开浏览器控制台查看错误信息

### 2. 声音断断续续
- 检查网络连接
- 关闭其他占用带宽的应用
- 尝试降低音频质量（修改 client.js 中的 SAMPLE_RATE）

### 3. 延迟太高
- 确保服务器在本地运行
- 检查 CPU 占用率
- 尝试使用有线网络而非 WiFi

### 4. 麦克风权限被拒绝
- Chrome: 点击地址栏左侧的锁图标 → 网站设置 → 麦克风 → 允许
- Firefox: 点击地址栏左侧的图标 → 权限 → 麦克风 → 允许
- Safari: 偏好设置 → 网站 → 麦克风 → 允许

## 性能指标

### 预期性能
- **延迟**: 50-150ms（端到端）
- **带宽**: 32kbps/用户（说话时）
- **CPU**: <5%（单用户）
- **内存**: <50MB（单用户）

### 监控方法
1. 打开浏览器控制台（F12）
2. 切换到 Network 标签
3. 筛选 WS（WebSocket）
4. 观察数据传输量

## 高级测试

### 1. 压力测试
打开 10+ 个窗口，所有人同时说话：
- 服务器应该能稳定运行
- 每个用户应该能听到所有其他用户

### 2. 网络模拟
使用 Chrome DevTools 模拟弱网环境：
1. F12 → Network 标签
2. 选择 "Slow 3G" 或 "Fast 3G"
3. 测试音频质量

### 3. 长时间运行
让通话持续 30+ 分钟：
- 检查内存泄漏
- 检查音频质量是否下降
- 检查 WebSocket 连接是否稳定
